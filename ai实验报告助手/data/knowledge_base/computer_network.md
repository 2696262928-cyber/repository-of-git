# 计算机网络课程知识库

本文件用于“计算机网络类实验报告”的 RAG 检索增强。内容参考 IETF RFC、Wireshark 用户指南等公开资料，并结合抓包、Socket、HTTP/TCP/UDP 实验改写为评分参考。它关注协议分层、报文字段、抓包证据和实验解释。

## 报告结构与评分关注点

计算机网络实验报告应包含：实验目的、网络环境、拓扑或通信对象、协议原理、实验步骤、抓包过滤条件、关键报文截图、字段解释、时序分析、结果验证和总结。评阅时重点看“是否能从抓包证据支撑结论”“是否解释协议字段含义”“是否区分应用层、传输层、网络层和链路层”“是否说明异常或丢包原因”。

高质量报告会写清楚客户端/服务器 IP、端口、协议、工具版本、过滤表达式、报文序号和时间戳。若报告只有 Wireshark 截图但没有字段解释，或只描述理论没有抓包证据，应提示补充。

## 分层模型、封装与地址标识

网络实验应说明数据在不同层次的封装关系：应用层数据交给传输层，传输层添加端口和可靠性相关字段，网络层添加 IP 地址，链路层添加帧头和 MAC 地址。分析报文时应区分 MAC 地址、IP 地址、端口号和应用层 URI/Host 等字段的作用。

评分时可关注：是否能解释同一通信中源/目的 IP 与源/目的端口；是否说明客户端临时端口和服务器固定端口；是否区分交换机基于 MAC 转发、路由器基于 IP 转发；是否理解 NAT、DNS、ARP 等辅助协议的作用。若把端口号当作进程号或把 MAC 地址当作公网地址，应提示概念错误。

## TCP 连接、可靠传输与拥塞/流量控制

TCP 是面向连接的传输层协议，核心特征包括序号、确认、重传、窗口和连接状态。实验中常见分析点包括三次握手、四次挥手、ACK、Seq、窗口大小、重传、乱序、RTT。RFC 9293 是现代 TCP 规范，可作为 TCP 字段和状态机的参考。

评阅时应检查：是否完整标出 SYN、SYN/ACK、ACK；是否解释序号和确认号如何推进；是否说明 FIN/ACK 的关闭过程；是否区分流量控制窗口和拥塞控制概念；是否用 Wireshark 的 Follow TCP Stream 或时序图辅助说明。若只写“三次握手保证可靠”而没有报文字段证据，应判为分析不足。

## UDP、DNS 与轻量报文通信

UDP 提供面向数据报的传输服务，报文头较小，不建立连接，也不提供 TCP 那样的可靠传输机制。它常用于 DNS、实时音视频、简单请求响应等场景。UDP 实验应说明端口、长度、校验和、请求和响应对应关系，并解释为什么某些应用选择 UDP。

评分时可关注：是否区分 UDP 的“无连接”和“不保证可靠”；是否展示 DNS 查询名、查询类型和响应记录；是否说明丢包、乱序或重试机制可能由应用层处理。若报告简单写“UDP 不安全”但没有结合实验现象，应提示表述过泛。

## HTTP 请求响应与状态码

HTTP 实验应说明请求方法、URI、Host、Header、状态码、响应头和响应体。RFC 9110 定义了 HTTP 语义，包括方法、状态码、字段和表示。常见实验包括 GET/POST、状态码 200/301/404/500、Cookie、缓存、内容类型、Keep-Alive 或 TLS 下的 HTTPS 抓包限制。

评阅时可检查：是否说明请求行和响应行；是否解释状态码类别；是否展示关键 Header；是否区分 HTTP 明文和 HTTPS 加密后可见字段减少；是否说明浏览器一次访问可能触发多个请求。若报告只有网页截图，没有网络请求证据，应提示补充抓包或开发者工具截图。

## Wireshark 抓包、过滤与字段解释

Wireshark 实验应写明捕获接口、抓包时间、过滤表达式和筛选理由。需要区分捕获过滤器和显示过滤器：捕获过滤器在开始抓包前减少采集数据，显示过滤器用于在已有数据中筛选显示。常见显示过滤包括 `tcp`, `udp`, `http`, `dns`, `ip.addr == ...`, `tcp.port == ...`。

评分时应关注：截图是否能看清报文编号、时间、源/目的地址、协议和摘要；是否展开关键字段；是否引用具体报文编号支持结论；是否避免用过宽过滤导致无关流量干扰。若截图只有包列表没有字段树和解释，分析深度不足。

## Socket 编程与网络实验验证

Socket 实验应说明客户端和服务器的角色、IP、端口、协议、阻塞/非阻塞行为、连接生命周期和异常处理。TCP Socket 通常需要 bind/listen/accept/connect/send/recv/close 等步骤；UDP Socket 通常使用 sendto/recvfrom 发送数据报。报告应展示服务端日志、客户端输出和抓包对应关系。

评阅时可检查：端口是否被占用时有处理；服务器是否支持多客户端；是否说明粘包/拆包或消息边界问题；是否处理连接关闭、超时和异常；是否证明发送的数据与抓包内容一致。若只展示程序输出而无协议分析，应提示补充网络层证据。

## 典型实验任务与检索关键词

常见计算机网络实验包括 Wireshark 抓包、HTTP 请求分析、DNS 查询、TCP 三次握手、UDP 通信、Socket 客户端服务器、Ping/ICMP、ARP、路由跟踪、简单 Web 服务器和网络性能测试。检索时可关注“抓包、过滤器、报文编号、IP、端口、Seq、Ack、SYN、FIN、HTTP 状态码、DNS 记录、Socket、粘包、超时”等关键词。

若报告出现“Wireshark、三次握手”，重点检查 SYN/SYN-ACK/ACK 报文编号和字段解释。若出现“HTTP GET/POST”，重点检查请求行、响应状态码和 Header。若出现“DNS”，重点检查查询类型和响应记录。若出现“Socket 聊天室/文件传输”，重点检查协议设计、消息边界、异常断开和抓包验证。

## 常见扣分点

- 没有说明网络环境、IP、端口和工具版本。
- 抓包截图模糊，无法看到字段。
- 只写协议理论，没有引用具体报文编号。
- 混淆 TCP 和 UDP 的可靠性、连接和报文边界。
- HTTP 实验只看网页效果，不分析请求响应。
- Wireshark 过滤条件错误，导致分析无关流量。
- Socket 实验没有异常处理和关闭流程说明。
