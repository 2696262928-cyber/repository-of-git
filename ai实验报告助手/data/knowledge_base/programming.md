# 程序设计课程知识库

本文件用于“程序设计类实验报告”的 RAG 检索增强。内容参考 Python 官方教程、PEP 8 等公开资料，并结合 C/Python/Java 等程序设计课程常见实验要求改写为评分参考。它只作为评阅依据，不能替代学生报告中的实际实现和运行结果。

## 报告结构与评分关注点

程序设计实验报告应至少包含：实验目的、问题描述、输入输出格式、算法或流程设计、关键数据结构、核心代码说明、测试用例、运行截图或输出、异常情况处理、结果分析和总结。评阅时重点看“需求是否被正确理解”“程序是否能运行”“测试是否覆盖正常与边界情况”“代码说明是否解释了设计原因”。只有代码而没有测试、没有输出、没有分析的报告，应判为完整性不足。

高质量报告会把题目拆成若干子任务，例如数据读取、参数校验、核心计算、结果输出、异常处理，并说明各部分之间的数据流。若实验是命令行程序，应写清楚输入示例和输出示例；若实验有文件读写，应说明文件路径、编码、异常场景和资源关闭方式。

## 问题分解、算法流程与伪代码

程序设计实验不能只粘贴最终代码，应先说明求解思路。常见写法包括：用自然语言描述算法步骤，用流程图表示分支和循环，用伪代码表达关键逻辑。若题目涉及查找、排序、统计、模拟、递归或动态规划，应说明为什么选择该算法，以及输入规模变大时可能的性能影响。

评分时可关注：报告是否区分“算法思想”和“语言语法”；是否说明循环不变量、递归出口、边界条件；是否解释关键变量含义；是否对复杂分支提供示例。若只有“本程序实现了某功能”这类泛泛描述，而没有可复现的步骤，应提示补充算法流程。

## 函数、模块与数据结构使用

程序应尽量把独立功能拆成函数或模块，而不是全部写在主函数中。函数设计应有明确职责、参数和返回值，避免过长函数、重复代码和大量全局变量。常见数据结构包括数组/列表、字典/映射、集合、字符串、结构体/类。报告需要说明为什么选用这些结构，例如字典适合按键快速查找，列表适合顺序遍历，集合适合去重。

评阅时可检查：函数名是否表达意图；参数是否过多；返回值是否被正确处理；数据结构是否与问题匹配；是否存在把所有状态都放进全局变量导致难以测试的问题。面向对象实验还应说明类的属性、方法、对象之间关系和封装边界。

## 输入输出、文件处理与异常处理

很多程序设计实验会因为输入假设过强而失分。报告应说明输入合法性检查，例如空输入、非数字输入、越界下标、文件不存在、编码错误、除零、数组越界等。异常处理不是简单捕获所有错误后忽略，而是要给出明确提示，并保证程序状态可恢复或安全退出。

文件处理实验应说明打开模式、编码、读写格式、关闭资源方式以及测试文件内容。若使用 Python，可关注 `with open(...)` 这类自动释放资源的写法；若使用 C/C++，应关注文件指针、返回值检查和缓冲区长度。若程序依赖外部文件，报告应给出相对路径或项目目录结构。

## 测试用例与结果验证

程序设计报告的测试不应只有一个成功样例。建议至少覆盖：典型输入、最小输入、最大或较大规模输入、空数据、非法输入、重复数据、边界值、异常路径。测试表可以包含“用例编号、输入、预期输出、实际输出、是否通过、说明”。若输出较长，可以只截取关键结果，但应保证可复现。

评阅时应检查测试是否真的对应题目需求。常见问题包括：只展示代码没有运行结果；测试输入和输出对不上；截图无法看清；只测正常路径；没有说明失败用例；把调试日志当作最终输出。高质量报告会解释失败原因和修复过程，而不是只展示最终正确结果。

## 代码质量、风格与可维护性

代码质量包括命名、缩进、注释、重复逻辑、模块划分、错误处理和可读性。PEP 8 等风格指南强调一致缩进、清晰命名和可读性；在其他语言中也应遵循课程或团队的代码规范。注释应解释“为什么这样做”，而不是逐行翻译语法。

评分时可关注：变量名是否有意义；魔法数字是否提取为常量；重复代码是否封装；复杂表达式是否拆分；是否存在未使用变量、死代码、硬编码路径；是否包含必要的运行说明。代码风格不是唯一评分点，但风格差通常会影响可维护性和调试效率。

## 典型实验任务与检索关键词

常见程序设计实验包括学生成绩管理、文件统计、字符串处理、矩阵运算、递归问题、日期计算、图书管理、命令行菜单、简单爬虫或数据清洗。评阅这些报告时应优先检索“输入输出、函数拆分、异常处理、文件读写、测试用例、边界值、运行截图、代码风格”等片段。

若报告出现“成绩排序、平均分、最高分、最低分、CSV、文本文件、菜单、增删改查”等关键词，重点检查文件格式、字段解析、非法输入和排序结果验证。若出现“递归、斐波那契、汉诺塔、回溯”等关键词，重点检查递归出口、状态回退和复杂度。若出现“类、对象、继承、封装”等关键词，重点检查类职责是否清晰、是否滥用全局状态。

## 常见扣分点

- 缺少实验目的、输入输出说明或运行环境。
- 只有代码，没有算法设计、测试结果和结果分析。
- 测试用例过少，未覆盖边界情况和异常情况。
- 代码截图代替源码，导致不可复制、不可复现。
- 输出截图模糊，或无法证明程序执行了题目要求。
- 异常处理缺失，输入稍有变化就崩溃。
- 过度依赖硬编码路径、固定数据或手工修改代码。
- 总结只写“学会了很多”，没有结合实验问题和改进方向。
