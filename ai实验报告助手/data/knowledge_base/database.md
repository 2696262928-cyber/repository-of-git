# 数据库课程知识库

本文件用于“数据库类实验报告”的 RAG 检索增强。内容参考 PostgreSQL 官方文档、数据库设计资料，并结合数据库课程常见实验改写为评分参考。它关注数据建模、SQL 正确性、事务一致性、索引与执行计划分析。

## 报告结构与评分关注点

数据库实验报告应包含：实验目的、业务背景、数据模型、表结构设计、约束设计、SQL 语句、执行结果、事务或并发分析、查询优化、测试数据和总结。评阅时重点看“表设计是否符合业务关系”“约束是否保证数据完整性”“SQL 是否正确返回目标结果”“结果截图是否可验证”“是否解释查询或事务现象”。

高质量报告通常会给出 ER 图、关系模式、主键/外键/唯一约束、样例数据、建表语句、查询语句和结果表。若报告只有 SQL 截图，没有说明表结构和测试数据，评阅时应提示其可复现性不足。

## 数据建模、范式与关系模式

建模实验应从实体、属性、联系和基数关系开始说明。关系模式需要明确每张表的主键、外键、非空约束、唯一约束和字段含义。范式设计的目标是减少冗余和更新异常，但也要结合查询需求考虑适度反规范化。

评分时可检查：是否把多值属性拆分成独立表；是否用外键表达一对多关系；多对多关系是否使用中间表；字段类型和长度是否合理；是否存在把多个含义混在一个字段中的问题。若主键只用业务名称而可能重复，应提示使用稳定标识或唯一约束。

## 约束、主键、外键与数据完整性

数据库实验中，约束是保证数据质量的重要机制。主键用于唯一标识记录；外键用于维护表之间引用关系；唯一约束、防空约束、检查约束可约束字段取值范围。报告应说明这些约束如何防止非法数据进入数据库。

评阅时应关注：外键是否引用主键或唯一列；删除或更新时是否说明级联、限制或置空策略；是否为频繁查询的外键列建立索引；是否展示插入非法数据时数据库拒绝的结果。若只在应用层检查而数据库层没有约束，应提示完整性保障不足。

## SQL 查询、连接、分组与聚合

SQL 实验应清楚说明查询目标、涉及表、连接条件、过滤条件、分组字段和排序规则。连接查询要注意 INNER JOIN、LEFT JOIN 等语义差异；聚合查询要说明 COUNT、SUM、AVG、MAX、MIN 的含义，以及 GROUP BY 和 HAVING 的使用场景。

评分时可检查：连接条件是否遗漏导致笛卡尔积；WHERE 和 HAVING 是否混用；聚合字段是否与分组字段匹配；NULL 值是否影响统计；排序是否保证输出稳定；查询结果是否与样例数据一致。复杂查询最好分步骤解释，避免只贴一条长 SQL。

## 事务、隔离级别与并发现象

事务实验应说明 ACID：原子性、一致性、隔离性和持久性。报告需要展示 BEGIN、COMMIT、ROLLBACK 的使用，并解释不同隔离级别下可能出现的脏读、不可重复读、幻读或序列化冲突等现象。PostgreSQL 官方文档说明其隔离实现与 MVCC 机制相关，不同数据库对标准隔离级别的实现可能不同。

评阅时可关注：是否使用两个会话复现实验；是否记录每一步 SQL 和观察结果；是否说明事务提交前后可见性变化；是否有回滚验证；是否解释锁等待或并发冲突原因。若报告只写“事务保证安全”，但没有实验过程，应判为分析不足。

## 索引、执行计划与性能分析

索引可以加速查询，但会增加写入和维护成本。数据库实验中应说明索引建在哪些列上、为什么这些列适合建索引、查询是否真的使用了索引。PostgreSQL 的 EXPLAIN 可用于查看查询计划，EXPLAIN ANALYZE 可观察实际执行信息，但会额外执行查询并产生测量开销。

评分时可检查：是否比较建索引前后的执行计划或耗时；是否区分顺序扫描和索引扫描；是否说明选择性、数据量、统计信息对计划的影响；是否避免给所有列盲目建索引。若只写“建索引后更快”但没有数据和执行计划支撑，应提示证据不足。

## 安全性、视图与权限

数据库应用实验还应关注安全性。应避免把用户输入直接拼接到 SQL 中导致注入风险；应说明账号权限、最小权限原则和敏感字段处理。视图可以封装复杂查询或限制用户看到的字段，但不能替代底层权限控制。

评阅时可关注：是否区分管理员、普通用户、只读用户；是否说明参数化查询；是否隐藏密码、手机号等敏感信息；是否对删除、修改操作提供确认或回滚方案。若报告截图中暴露真实账号密码，应提示安全问题。

## 典型实验任务与检索关键词

常见数据库实验包括 ER 图设计、建表与约束、数据插入更新删除、单表查询、多表连接、分组聚合、视图、存储过程、触发器、事务隔离、索引优化、执行计划分析和简单数据库应用。检索时可关注“主键、外键、唯一约束、非空约束、JOIN、GROUP BY、HAVING、事务、隔离级别、索引、EXPLAIN、SQL 注入”等关键词。

若报告出现“学生选课、图书借阅、订单管理、库存管理”等业务场景，重点检查实体关系、外键和多对多中间表。若出现“事务、并发、转账”，重点检查两个会话、提交/回滚和隔离现象。若出现“索引、慢查询、优化”，重点检查数据规模、执行计划和前后对比。若出现“登录、注册、权限”，重点检查参数化查询和敏感数据保护。

## 常见扣分点

- 没有 ER 图、关系模式或表结构说明。
- 建表语句和查询结果不可复现，缺少样例数据。
- 主键、外键、非空、唯一等约束缺失。
- SQL 查询没有解释连接条件，结果可能来自笛卡尔积。
- 事务实验没有两个会话过程，无法体现隔离性。
- 索引优化只有结论，没有 EXPLAIN 或耗时对比。
- 截图过多但缺少文字分析。
